# Track Specification: Advanced Request Processing Hardening (v0.51.0)

## 1. Overview
The basic structures for `map`, `split_clients`, `mirror`, and `auth_request` exist. However, the system completely lacks a Variable Evaluation Engine. Because NO variables can be resolved dynamically during a request, `map` and `split_clients` are effectively dead code, and dynamic routing (e.g., `proxy_pass = "$backend"`) is impossible. Additionally, the traffic mirroring logic is written but never invoked in the main request handler.

This track focuses on building the central `VariableContext` to evaluate Nginx-style variables on the fly, intercepting the auth subrequest to inject upstream headers, and spawning the async mirror tasks.

## 2. Functional Requirements

### 2.1 Variable Evaluation Engine
- Implement a `VariableContext` struct created per HTTP request in `http_proxy::handle_request`.
- The engine must lazily resolve variables like:
    - Built-in: `$host`, `$uri`, `$remote_addr`, `$request_method`.
    - HTTP Headers: `$http_<name>` (e.g. `$http_user_agent`).
    - Configured bindings: Values generated by `map` and `split_clients`.
- Update `proxy_pass` resolution: instead of using the raw string, it must pass through the `VariableContext` to substitute variables (e.g. `http://api-$region.internal`).

### 2.2 Traffic Mirroring Execution
- In `http_proxy::handle_request`, before proxying to the main backend, check if a `mirror` directive is configured for the matched location.
- If `mirror_percentage` allows, clone the `hyper::Request` parts and the body (safely buffering or stream-teeing if `mirror_request_body = true`).
- `tokio::spawn` a background task that sends the cloned request to the mirror URL using `hyper::client`.
- The main request MUST NOT wait for the mirror request to complete.

### 2.3 Auth Request Header Forwarding
- Update `auth_request.rs` so that if the authentication service returns a 2xx response, it parses headers matching `auth_request_set`.
- Inject these captured headers into the `VariableContext` so that `proxy_set_header` can forward them to the primary backend.
- Implement an in-memory TTL cache for auth requests (using the `Authorization` header or request signature as a key) to avoid overwhelming the auth service.

## 3. Non-Functional Requirements
- **Zero-Copy Laziness:** `VariableContext` should heavily use standard rust references `&str` and `Cow<str>` to avoid String allocations until a variable is actually requested.
- **Resilience:** If the mirror backend is down, the background task should gracefully log the error and exit without affecting the primary stream.

## 4. Acceptance Criteria
- [ ] A `map` directive checking `$http_user_agent` correctly sets a `$backend` variable, and `proxy_pass = "$backend"` routes the request correctly.
- [ ] `split_clients` correctly assigns an A/B bucket based on `$remote_addr`.
- [ ] A configured `mirror` duplicates exactly the specified percentage of HTTP POSTs, including their bodies, to a secondary server without slowing down the primary response.
- [ ] `auth_request_set` successfully captures an `X-User-Id` from the auth subrequest and injects it into the backend request.
